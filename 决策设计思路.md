## 决策设计思路

你的 Agent 测试时表现完美，上线后时好时坏。你改了无数版 prompt，每次都像打地鼠——修好一个问题，另一个问题又冒出来。

问题不在 prompt 写得不够好。问题在你的 Agent 架构本质上是一个「凭直觉综合判断的面试官」——而认知科学花了七十年证明，这是人类已知最不可靠的判断模式。这篇文章从噪声分析、偏差-方差分解、结构化判断三个层面，推导出 12 条 Agent 架构设计原则。不教你用任何框架，但帮你理解一个更根本的问题：**怎么设计一个让错误更难发生的系统。**

---

## 1. 一个你肯定经历过的场景

你做了一个 Agent。它能搜索、能调用工具、能根据检索结果综合回答。

测试的时候表现很好。上线。

第一天用户反馈：「它有时候挺聪明的，有时候突然很傻。」你查日志——同一个问题，问法稍微变一下，Agent 走了完全不同的判断路径。同一个工具调用，这次对了，那次莫名其妙地没触发。

你改了一版 prompt。好了一周。然后又有人反馈新问题。

你开始怀疑：是模型不够强？是该升级到最新版？是该加更多 few-shot？

都不是最根本的问题。

**最根本的问题是你把判断当成了一件可以「用更好的指令来解决」的事情，而不是一件需要「设计判断流程」的事情。**

为了说清楚这一点，我们需要先做一个诊断。

---

## 2. 噪声审计：你的 Agent 到底在哪一层抖？

1960 年代，有人对 208 位美国联邦法官做了一项实验。研究者设计了 16 起虚构案件，请所有法官分别给出量刑。

结果令人不安：同一起案件，不同法官给出的刑期差异巨大。单起案件中最极端的情况下，有法官判了 0.5 年，有法官判了 25.5 年。

如果把这些差异拆开，会看到三层结构：

- **水平噪声**：法官A平均判6.6年，法官B平均判8.4年——有人整体偏严，有人整体偏宽。
- **模式噪声**：法官A总体偏严，但对白领犯罪格外宽容；法官B总体偏宽，但对惯犯格外严厉——每个人对不同案件类型有自己特定的「偏见组合」。
- **情境噪声**：同一个法官，临近中午时更可能拒绝假释（饿了）；当地足球队周末输了球，周一判决更严厉——随机情境波动。

如果我们把这个框架平移到你的 LLM Agent 上，会发现几乎完美对应。

### 2.1 水平噪声

你的 Agent 的「法官整体倾向」来自哪里？

| 噪声源        | 表现                                                         |
| ------------- | ------------------------------------------------------------ |
| 模型选择      | GPT-4 遇到模糊问题时倾向于展开解释，Claude 倾向于先追问澄清。换一个模型，整个 Agent 的行为基线就漂了。 |
| System Prompt | 「你是严谨的专家」和「你是热心的助手」——同一个 query，路由判断、回复风格、甚至是否调用工具的倾向都不同。 |
| Temperature   | 高 temperature 意味着决策分布在更广的空间里——等价于你的「法官」群体内部的严厉程度方差变大了。 |

每一次你调参、换模型、改 prompt，都在不知不觉中改变 Agent 的判断基线。这不是优化，这是水平噪声的漂移。更麻烦的是，你不是在一个可控实验中观察它——你是直接在线上改的。

### 2.2 稳定模式噪声（这一项最大）

卡尼曼梳理了几十年研究后发现一个一致的规律：**稳定模式噪声几乎总是最大的噪声来源。** 水平噪声和情境噪声加起来都比不上它。

在 Agent 中，稳定模式噪声长这样：

- **路由偏见**：同一个 LLM，对包含数字的 query 系统性地倾向于调用计算器——哪怕用户只是问「iPhone 15 和 16 有什么区别」。
- **权威格式偏见**：LLM 对格式良好的、看起来专业的信息系统性地降低怀疑——「排版漂亮的假新闻」比「排版粗糙的真新闻」更容易被判定为相关。
- **沉没成本污染**：Agent 自己生成了一个搜索 query，然后用同一个 LLM 调用来评估搜索结果的相关性。评估的准确性被污染了——因为你已经投入了认知成本去生成这个 query，你天然倾向于认为搜回来的东西是相关的。这是人类面试官的经典偏误：对一个自己起草了面试问题的候选人，打分时更宽容。
- **近因效应**：上下文窗口中越靠后的信息，对判断的权重越大。用户刚说完一句情绪化的话，接下来的路由判断就会受影响——不是因为那句话真的应该改变路由逻辑，而是因为它恰好是模型注意力窗口里最新的东西。

这些不是随机波动。它们是对特定**任务-情境组合**的可复现的、系统性的偏差。而且——重点来了——**这些问题不能通过「更好的 prompt」解决。**

为什么？

因为 prompt 的作用是给出判断指令。但稳定模式噪声恰恰发生在「模型按照你的指令认真判断」的过程中——它认真地理解你的指令，认真地在不同维度上分配权重，然后因为嵌入空间中的某些系统性倾斜（数字→计算器、权威排版→可信）而做出了系统性的偏移。

你改 prompt，往往是在和模型的既有倾向**对抗**。这是一种注意力竞争——你在 prompt 里说「注意，不要看到数字就想调用计算器」，模型需要消耗注意力来遵守这条指令，而注意力是稀缺资源。下一次它可能记住了这条指令，但放松了另一条——「应该主动追问」那条被挤压了。

更好的 prompt 不是解药。**把判断拆开，让每个判断在更干净的条件下独立完成，才是。**

### 2.3 情境噪声

这是被讨论最多的一项，因为它的表现最直观——「同样的 prompt 运行两次，输出结果不一样」。

但情境噪声在 Agent 中有一个更隐蔽的来源：**上下文窗口内的信息排列顺序。**

你不是在设计一个回答问题的模型。你是在设计一个**决策管道**。管道中每一步的输出都将成为下一步的输入——而输入的先后顺序会系统性地影响判断。这不是 bug，这是 LLM 注意力机制的结构性特征。

### 2.4 语义噪声（LLM 独有）

两个人类法官对「严厉」的理解可能不同，但至少他们对「7年有期徒刑」的物理含义有共识。LLM 没有这个锚定。

你让 Agent 评估搜索结果的相关性，「1-5 分」。在上下文 A 中，「相关性=4」的含义是；在上下文 B 中，因为前面的检索结果整体质量更高或更低，「相关性=4」的含义悄悄漂移了。同一个分数，在不同的上下文中不是同一把尺子。

这叫**语义噪声**——评分标准本身的上下文条件化漂移。人类可以通过行为锚定（BARS）来缓解——不是「领导力1-5」，而是「领导力-3分：能在会议中清晰陈述团队目标；领导力-7分：能在冲突中重新框定问题使双方达成共识」。但对于 LLM，行为锚定本身也受上下文影响——锚定描述也是文本，也会在不同上下文中漂移。

这是一个深度问题。目前能做的不是消除，而是通过架构约束来限制它：**用独立调用 + 短上下文评估，减少上下文污染的范围。**

## 3. 核心诊断：你的 Agent 是个「凭直觉的面试官」

1954 年，心理学家保罗·米尔出版了《临床预测 vs 统计预测》。他梳理了几十个研究，发现一个反直觉的结论：

**专家对候选人表现的综合判断，与实际绩效的相关系数只有 0.15。而一个简单的统计回归公式——挑几个客观指标，算加权总分——相关系数是 0.32。**

0.15 是什么概念？只比抛硬币好一点点。

0.32 是什么概念？好了一倍还多。

此后七十年，这个发现在招聘、医疗诊断、信用评估、项目评审等几十个领域被反复验证。不管专家经验多丰富、面试流程多复杂、笔试题目多精巧——临床判断始终打不过机械预测。

**不是因为专家不聪明。是因为专家判断中的噪声太大——尤其是稳定模式噪声。** 你花了多少心思去了解候选人的特殊情况，模型看来根本不特殊。你的「具体情况具体分析」在统计意义上等价于「在每个案例中加入一个新的噪声项」。

现在，让我们诚实地审视一个典型的 LLM Agent 架构：

```
用户 Query
    ↓
[一个巨大的 System Prompt，包含了角色描述、工具列表、
评估标准、回复格式要求、安全规范、边缘情况处理指南...]
    ↓
LLM 一次性判断：需要搜索吗？搜什么？结果相关吗？
用哪些结果？回复多长？要不要追问？
    ↓
输出
```

这个架构，用决策科学的术语来说，是**纯临床判断**。所有信息塞进一个上下文，让模型「综合考量」。模型确实在认真考量——它在注意力机制中分配权重，在不同需求之间做权衡。但问题恰恰出在这里：**注意力竞争本身就是一个巨大的噪声来源。**

你花了心思写了那段「请仔细评估搜索结果的相关性，不要被排版和权威语气影响」——但这段指令在 prompt 里只是 2000 个 token 中的一个段落。模型在执行时，它被淹没在角色描述、工具规范、格式要求和用户 query 的背景噪声中。它在努力平衡，而平衡的结果就是：有时候平衡得好，有时候平衡得不好。后一种情况，你称之为「Agent 又抽风了」。

这不是抽风。这是**用单一 LLM 调用承担多步判断的必然结构性问题。**

## 4. 一个公式，说清一切

在数学上，这个问题可以精确地表达。

任何一个判断结果 \(\hat{\theta}\) 偏离真实值 \(\theta\)，其总体误差可以分解为：

\[\mathbb{E}[(\hat{\theta} - \theta)^2] = (\mathbb{E}[\hat{\theta}] - \theta)^2 + \text{Var}(\hat{\theta})\]

左边是**均方误差**（MSE，总体错误）。右边第一项是**偏差**的平方——你的判断系统性地瞄歪了多少。右边第二项是**噪声**（方差）——你的判断瞄得有多散。

这个分解的重要性怎么强调都不为过：**偏差和噪声对总体错误的贡献是加性的，而且是同等权重的。** 降低偏差和降低噪声，在减少总体错误上一样重要。

但 Agent 开发的现状是什么？**绝大多数调优动作都在降偏差。** 你觉得 Agent 的回复不够准确，你改 prompt 让它更「认真」、加更多规范、更多 few-shot——你在试图降低系统偏差。但你几乎没有在降噪声。

而且更糟的是，你降偏差的一些操作，可能在**增加噪声**。你加了一条指令——「注意区分用户是在问事实还是在寻求建议」——这条指令本身就是一个新的需要模型分配注意力去遵守的约束。更多约束 → 更拥挤的注意力竞争 → 更高的噪声。你在用一个同时增加噪声的手段去降低偏差。净效果不一定是正的。

这就是为什么 prompt 优化有天花板。那个天花板不是模型能力的上限，而是**单一 LLM 调用中判断噪声的下限。**

## 5. 决策科学的三个核心教训

在进入具体设计原则之前，我们需要先建立几个来自决策科学的锚点概念。你会发现它们每一个都能几乎逐字翻译成 Agent 的架构约束。

### 5.1 谷歌结构化面试：不是更好的面试官，而是更好的面试流程

谷歌曾经和所有公司一样，依赖传统的面试——一群面试官轮流聊，各自凭直觉打分，最后讨论决定。效果呢？也是那个 0.15。

于是谷歌重新设计了面试流程。整个设计的底层逻辑只有三个词：**分解、独立、聚合。**

**第一步：分解。** 谷歌选了四个指标：认知能力、领导力、「Googleyness」（这个人的气质是否契合谷歌）、专业知识。只选四个，少而精，互不重叠。然后——这是关键——**每一轮面试只考察一个指标。** 面试官被明确要求：不要考虑指标之外的因素。候选人的外表、谈吐、兴趣爱好，既不提问，也不纳入评估。

**第二步：独立打分。** 每位面试官只给自己负责的那个指标打分。不允许打听其他指标的分数。面试官的任务不是「判断我喜不喜欢这个人」，而是「依据统一的评分标准，尽可能客观地给这个指标打分」。

**第三步：综合判断。** 四个指标的分数全部摊在桌面上。委员会综合讨论，做出最终决定。但注意：讨论的目的不是「再来一轮直觉判断」，而是检查打分过程是否有遗漏、有错误。分数是基础，讨论是校准，不能推翻分数另起炉灶。

结果呢？虽然没有公布精确的相关系数，但谷歌的内部评估表明，这套流程显著提高了招聘质量。而且不只是谷歌——结构化面试的有效性在此后被无数组织重复验证。

现在，把这三个步骤映射到 Agent 架构上——你看到的不是三个步骤，而是三个架构原则：

| 谷歌面试                                 | Agent 架构                                                  |
| ---------------------------------------- | ----------------------------------------------------------- |
| 四个独立指标                             | 每个决策维度独立调用，独立 prompt                           |
| 面试官不知道其他分数                     | 每个 LLM 调用只知道它需要知道的上下文，看不到其他维度的结果 |
| 委员会基于分数讨论，而非推翻分数重新判断 | 聚合层用确定性规则或加权公式，而非另一个 LLM「综合考量」    |

谷歌没有发明什么魔法。他们只是把一个「凭直觉综合判断」的流程，改造成了一个「分步结构化判断」的流程。面试官没变，候选人没变，变的只是判断的结构。而判断结构的改变，足以把决策质量从 0.15 拉到更高的区间。

**你的 Agent 就是谷歌改革之前的面试流程。改 prompt 就像换一个面试官。而真正需要的是重组整个面试流程。**

### 5.2 法官的噪声：为什么「综合考量」是最危险的词

回到法官量刑实验。

卡尼曼把 208 位法官的数据做了方差分解后发现：系统噪声（水平噪声+稳定模式噪声+情境噪声）占总变异的大部分。而且——**稳定模式噪声几乎总是最大的一项。**

这个发现的深层含义是：人不只是在「整体严厉还是宽松」上有差异（水平噪声）。人更大的问题是，**每个人对不同类型的案件有一套独特的、高度个人化的「偏误指纹」**——某位法官对白领犯罪心软，某位法官对惯犯格外严厉，某位法官对年轻被告容易产生共情。

这不是他们不专业。这是「综合考量」的必然产物。当一个判断需要综合大量信息、在不同维度之间分配权重，而权重分配本身没有客观标准——不同的人就会稳定地、可复现地分配出不同的权重。这就是稳定模式噪声的来源。

LLM 有没有「偏误指纹」？

当然有。它比人类更隐蔽，但也更可复现。GPT-4 对权威语气的文本有一种难以消除的信任倾向。Claude 在面对伦理困境时比 GPT-4 更倾向于回避给出明确立场。Gemini 在面对数值问题时比其他模型更容易过度自信。这些不是随机波动——它们是模型训练数据、RLHF 对齐策略和架构特性共同塑造的稳定模式噪声。

而且，LLM 的模式噪声还有一个额外的特性：**它可以在一次调用中同时激活多套模式噪声。** 你让 Agent 在一个 prompt 里同时判断「是否需要搜索」「搜索什么」「结果相关吗」「该怎么回复」，这四个判断中的每一个都激活了各自的一套隐式权重分配。它们之间互相干扰、互相污染。最终表现出来的，就是你看到的「有时候聪明有时候傻」。

谷歌的结构化面试解决的就是这个问题——**通过分解和独立打分，把模式噪声的通道一个一个地切断。** 面试官A只打认知能力的分数，他在「白领犯罪宽容度」上的个人偏好就完全没有机会影响他对认知能力的判断。不是因为他克服了偏见，而是因为偏见的通道被架构阻断了。

### 5.3 决策卫生：好制度，不是好人

卡尼曼在《噪声》中提出了一个有力的类比：**决策卫生。**

手术室为什么比一百年前安全得多？不是因为外科医生更聪明了，而是因为无菌操作规范、术前 checklist、术后感染监控——这些**制度设计**让一个疲惫的、精力不集中的医生也不容易感染病人。

你不需要医生是一个「极度自律、绝不出错」的超人。你需要的是：即使医生今天状态不好，制度和流程也能兜底。

**决策卫生 = 决策领域的无菌操作规范。**

它的核心心态是：**看不见的病菌会感染伤口 → 看不见的噪声会污染判断。** 你不依赖判断者的自觉来洗手，所以你也不应该依赖 LLM 的自觉来遵守 prompt 中的行为规范。

这就是为什么「把规范写在 prompt 里」和「把规范外化为架构约束」之间有质的区别：

| Prompt 中的规范                  | 架构中的约束                                                 |
| -------------------------------- | ------------------------------------------------------------ |
| 「请仔细检查回复中的事实准确性」 | 独立的 Verifier 模块，逐条交叉比对事实与检索结果，不通过则自动退回重生成 |
| 「如果涉及医疗建议请务必谨慎」   | Router 识别医疗 query→自动附加免责声明+限制回复可触达的工具集 |
| 「回复保持简洁，不超过200字」    | 后处理阶段截断+压缩，不依赖 LLM 在生成时自我限制             |

prompt 中的规范 = 要求医生「记得洗手」。架构中的约束 = 把洗手池放在手术室入口，不洗手不开门。

前者依赖当下这一刻 LLM 的注意力分配（而注意力是竞争性的、有限的、受情境噪声影响的）。后者不经过 LLM 的注意力系统——它在 LLM 之外执行。

**决策卫生是本文所有设计原则的哲学底座。** 每一条原则本质上都在做同一件事：把判断质量从「LLM 当时的状态好不好」中解放出来，交给一个即使 LLM 状态不好也能兜底的系统架构。

---

## 6. 12 条设计原则

以下是基于前述理论推导出的 12 条 Agent 架构设计原则。它们不绑定任何具体框架，也不承诺「遵循这些原则 Agent 就完美了」。它们回答的是一个更有限但也更诚实的问题：**如果你想减少 Agent 判断中的噪声，你应该把架构设计成什么样？**

---

### 原则 1-4：架构骨架

这四条定义 Agent 判断流程的基本结构。没有这四条做基础，后面的评估、验证、监控原则就无从谈起——你连判断是什么时候做的、谁做的都说不清。

---

#### 原则 1：分解 —— 每个决策点拆为独立调用，单一职责

**当前典型反模式**：

```
[System Prompt]
你是一个智能助手。当用户提问时，你需要：
- 判断问题是否需要搜索
- 如果需要，生成合适的搜索 query
- 评估搜索结果的相关性和质量
- 根据相关结果生成回复
- 如果信息不足，决定是否追问
请综合以上步骤，一次性完成回复。
```

这个 prompt 的问题不在于写得太长。问题在于它把 Router、Searcher、Evaluator、Responder 四个角色塞进了同一个 LLM 调用里。它们共享同一个注意力预算、同一个上下文窗口、同一套隐式权重分配。

当 LLM 在生成搜索 query 时，它已经看到了自己即将要综合回复的上下文。这意味着：
- 如果 query 生成偏了，评估结果的步骤倾向于「宽容」（沉没成本效应：我为这个 query 花了注意力，搜回来的东西应该有用）
- 如果评估结果偏了，回复生成的步骤倾向于「硬用」不相关的信息（衔接偏误：我已经写了这些搜索结果在上下文里，我必须让回复和它们衔接上）

**修正架构**：

```
[Router]      → 仅判断：这个问题需要搜索吗？
                 输入：用户 query + 核心上下文摘要
                 输出：{need_search: bool, confidence: float}

[Searcher]    → 仅执行：生成搜索 query（不知道 Router 的判断逻辑）
                 输入：用户 query + Router 的 need_search=true 信号
                 输出：{queries: [...]}

[Evaluator]   → 仅判断：每条搜索结果的相关性 1-5
                 输入：用户 query + 单条搜索结果（不知道 query 是怎么生成的，不知道其他结果的内容）
                 输出：{relevance: int, rationale: str}

[Aggregator]  → 仅综合：基于独立评分 + 预设规则决定用哪些结果
                 输入：所有结果的 {id, relevance} 键值对
                 输出：{selected_ids: [...], selection_rule: "threshold_4"}
                 **不要用 LLM 做这一步。这是规则引擎。**

[Responder]   → 仅生成：基于已筛选的结果写回复
                 输入：用户 query + 选中的搜索结果（不知道被筛掉的结果）
                 输出：回复文本
```

**关键设计约束**：
- 每个模块只接收它**需要**的信息，不接收更多。Searcher 不需要知道 Evaluator 的评分标准。Evaluator 不需要知道 Router 的判断逻辑。
- 模块之间的信息传递是**结构化数据**（JSON），不是自由文本。这切断了自然语言中的语义噪声污染——一个模块不能通过措辞的细微差异影响下一个模块的判断。
- 每个模块的 prompt 是**独立且简短的**。不共享一个巨大的 system prompt。每个模块的 prompt 只包含它这一个职责需要的指令 + 行为锚定的评估标准。

**成本认知**：这种架构增加 LLM 调用次数。是的。但需要区分两种成本：计算成本（多几次 API 调用）和错误成本（一次「综合判断」中噪声导致的错误决策）。对于高风险的决策点（医疗、金融、法律），错误成本远高于计算成本。对于低风险决策点（闲聊回复的语气微调），可以缩小分解粒度。这是一个**风险-架构精度**的梯度设计，不是全有或全无。

---

#### 原则 2：独立打分 + 规则聚合 —— 评估维度各自独立，聚合不用 LLM

这是原则 1 在「评估」场景下的特定展开。

当你需要 Agent 在多个维度上评估一个候选（候选回复、搜索结果、工具调用方案）时：

**❌ 不要这样**：

```
请评估以下三个候选回复，从准确性、完整性、语气三个维度评分，选出最好的一个。
```

看起来是结构化的——三个维度。但问题在于：LLM 在评估「准确性」时，已经看到了另外两个维度的要求、候选回复的整体面貌、以及它自己即将要做「综合选优」的预判。这些信息互相污染。

**✅ 要这样**：

```
Step 1（独立调用，不知道其他维度）:
[Accuracy Evaluator] → 仅评估事实准确性 → {score: 4, rationale: "..."}

Step 2（独立调用，不知道 Step 1 的结果）:
[Completeness Evaluator] → 仅评估信息完整性 → {score: 3, rationale: "..."}

Step 3（独立调用，不知道前两步的结果）:
[Tone Evaluator] → 仅评估语气适当性 → {score: 5, rationale: "..."}

Step 4（非 LLM）:
[Aggregator] → 加权公式: total = 0.5*accuracy + 0.3*completeness + 0.2*tone
             → 选最高分
```

**为什么聚合层不能用 LLM？**

因为你一旦把三个分数交给 LLM 去做「综合判断」，你在前面三步苦心分离的噪声通道全部重新接通了。LLM 会用自己的隐式权重重新解读那些分数——「accuracy=4 但 completeness=3，我觉得 completeness 更重要...」——而这恰恰就是你在第一步试图消除的稳定模式噪声。你花了三步的 API 调用成本来降噪，第四步一招打回原形。

**规则聚合的替代方案**：

如果「加权求和」太粗糙，可以使用更复杂的确定性规则：
- **否决规则**：任一核心维度低于阈值 → 自动淘汰，不走聚合
- **Pareto 前沿**：只在互有优劣的候选中触发更重的评估流程
- **多轮淘汰**：先用最便宜的维度筛掉 80%，再用贵的维度精细比较剩下的

核心理念不变：**聚合逻辑是开发者定义的，不是 LLM 临场「权衡」出来的。**

---

#### 原则 3：机械预测优先 —— 能用规则就不用模型，能用小模型就不用大模型

不是所有判断都需要 LLM。

| 层级              | 技术                          | 示例                                               | 噪声水平                     | 适用场景                     |
| ----------------- | ----------------------------- | -------------------------------------------------- | ---------------------------- | ---------------------------- |
| L1 确定性规则     | if-else, regex, lookup        | 「hello」「谢谢」→ 不需要搜索                      | 零噪声                       | 高频、确定性强的判断         |
| L2 统计/小模型    | 分类器, embedding相似度, BM25 | 判断 query 是否包含时效性词汇                      | 低噪声（偏差可能，噪声极小） | 模式稳定、数据充足的判断     |
| L3 LLM 结构化判断 | 独立维度 + 锚定标准           | 判断检索结果是否直接回答用户问题                   | 中噪声（可控但存在）         | 需要语义理解的判断           |
| L4 LLM 综合判断   | 真正的「理解」                | 判断两个同样合理但侧重不同的回复哪个更适合当前用户 | 高噪声                       | 无法拆解为独立维度的开放判断 |

**核心理念**：每个决策点，尽可能用 L1-L2 解决，L3 作为兜底，L4 只在真正需要时使用。

每往下一层，引入的噪声增加约一个数量级。好的 Agent 设计就是把尽可能多的判断留在上层。

**实例**：判断用户 query 是否需要搜索。

- **L1 方案**：维护一个「不需要搜索」的模式库——纯闲聊、简单常识、对自身能力的询问。直接匹配。覆盖 60% 的 query，零延迟，零噪声。
- **L2 方案**：训练一个轻量分类器——基于 query embedding + 历史标注。覆盖率推到 85%。
- **L3 方案**：LLM 按结构化标准逐条判断（「这条 query 涉及时效性信息吗？」「这条 query 中的事实声称超出常识范围吗？」）。覆盖剩余 15%，有噪声但可控。
- **L4 方案**：直接让 LLM 凭直觉说 yes/no。

大多数当前 Agent 把所有 query 都推给了 L4——不是因为 L1-L3 做不到，而是因为 L1-L3 需要额外的工程投入。而「prompt 里加一句话」是更容易的路径。容易，不等于正确。

---

#### 原则 4：显式停止规则 —— 强制终止「再搜一次可能更好」的无限循环

当前 Agent 最不优雅的行为之一：搜索 → 对结果不满意 → 换 query 再搜 → 还是不满意 → 再换 → ...

这不是 Agent 的错。这是**最优停止问题**没有被解决。

秘书问题的数学告诉我们：继续搜集信息的边际收益递减。多搜一次，期望的信息增量是正的，但增量本身在不断缩小。如果每次搜索有成本（时间、token），那么在某个点，停止搜集、用现有信息做出判断，是**期望效用最优**的选择。

但 LLM 不知道这一点。它天然倾向于「再查一下」。因为：
- 生成新 query 的成本对它来说是零（它不付 API 费）
- 它的训练数据中，「更全面」通常被奖励

**每个 Agent 的决策循环需要内置硬编码的停止规则**：

```
FORCE_STOP if ANY of:
  - 连续 N 次搜索结果无新增信息（信息增益 < 阈值 ε）
  - 从第一次搜索起已耗时 > T_max
  - 工具调用总次数 > N_max
  - 用户等待时间 > T_user_max → 先返回部分结果 + "继续查找中"
  - Evaluator 返回 "所有必要信息已获取"

ON_FORCE_STOP:
  - 用当前最佳信息回复
  - 明确标注「以下信息可能不完整」
  - 提供「继续深入」的选项给用户
```

停止规则不是让 Agent「更高效」——是让 Agent 的决策循环**在数学上有界**。一个没有上界的决策循环不是工程系统，是赌博。

---

### 原则 5-8：质量保证

这四条定义 Agent 如何验证自己的输出。不依赖「让模型再检查一遍」。

---

#### 原则 5：决策日志 —— 让判断过程成为可审计对象

当前 Agent 的行为像一个黑箱。query 进去，回复出来。中间发生了什么，你无从得知。出了问题，你能做的只有：猜。猜是哪个步骤出了问题。猜应该改哪段 prompt。

这不仅低效，而且从决策科学的角度看，这是**剥夺了自己复盘的能力**。该讲中提到，「复盘应该看过程不看结果」——但如果过程没有被记录，你就没有东西可以复盘。

每个关键决策点，Agent 应该输出**结构化日志**，而非自由文本：

```json
{
  "decision_id": "route_d854f3",
  "decision_type": "tool_selection",
  "timestamp": "2026-04-21T14:32:11Z",
  "context_snapshot": {
    "user_query_hash": "a3f8c...",
    "conversation_turn": 7,
    "available_tools": ["search", "calculator", "database_query"]
  },
  "independent_scores": {
    "need_external_info": 0.87,
    "need_computation": 0.12,
    "need_structured_data": 0.03
  },
  "decision": "search",
  "decision_rule": "max_score > 0.7",
  "alternatives_considered": ["calculator"],
  "reason_for_exclusion": "score 0.12 below threshold 0.7",
  "latency_ms": 234
}
```

这个日志的价值是**三重**的：

**第一，离线评估。** 你可以事后标注：在这些情境下，Agent 选择了搜索，但实际上直接回答才是最优的。然后观察：这些错误情境下，`need_external_info` 的分数是否系统性地偏高？如果是，你发现了一个**稳定模式噪声**——LLM 对某类 query 系统性地高估了「需要外部信息」的概率。这比「Agent 有时候该搜不搜、不该搜乱搜」的模糊直觉精确得多。

**第二，在线监控。** 如果 `need_external_info` 的分数分布突然从均值 0.3 跳到 0.7——这不是 bug，这是你的 Agent 的行为基线在无声地漂移。可能因为模型提供方更新了版本，可能因为用户群体的 query 分布变了，也可能因为你的某次 prompt 微调产生了意外副作用。**在你收到用户投诉之前，你已经看到了异常。**

**第三，对抗性审计。** 如果有人试图通过 prompt injection 让 Agent 调用不该调用的工具，可能在最终行为上体现为一次「正常的」错误决策——但在日志的结构化分数上，你会看到异常模式：`need_external_info` 得分极高但 query 的语义特征与高得分不匹配。攻击在被执行的路上留下了指纹，即使单次行为看起来正常。

---

#### 原则 6：独立验证器 —— 不是「再查一遍」，是「换人查」

当 LLM 生成了回复之后，最常见的质量保障手段是——让同一个 LLM（或同一个模型的另一个实例）再检查一遍生成的回复，判断是否有问题。

这被称为**自验证**。它的问题在于：**噪声高度相关。** 生成时犯的错，和验证时检查的错，可能源于同一套隐式权重分配。如果 GPT-4 对「权威格式的信息」有信任偏误，这个偏误在生成阶段让它错误地采纳了一条不实信息，在验证阶段也同样会让它倾向于判定这条信息「看起来没问题」。

自验证的有效性被系统性地高估了。

**有效的验证需要结构**：

```
[Generator]      → 生成候选回复

[Verifier-A]     → 仅验证事实准确性
                   输入：候选回复 + 原始检索结果（不知道 Generator 的 prompt）
                   输出：{claim: str, verdict: "supported"|"contradicted"|"unverifiable",
                          evidence: str}

[Verifier-B]     → 仅验证推理链条连贯性
                   输入：候选回复（不知道原始检索结果，不知道 Verifier-A 的结果）
                   输出：{logical_gaps: [...], overall_coherence: 1-5}

[Arbiter]        → 规则引擎：
                   if Verifier-A 有任何 "contradicted" 或 Verifier-B coherence < 3:
                     → 标记 + 退回重生成（或直接回复 + 不确定性声明）
                   else:
                     → 通过
```

**关键约束**：
- Verifier 的 prompt 不能和 Generator 的 prompt 相似（避免同一套隐式权重复用）
- 两个 Verifier 看到的信息是不同子集（进一步降低噪声相关性）
- Arbiter 是规则引擎，不做判断，只执行预设的判定逻辑

**成本权衡**：并不是每个回复都需要双验证器。对低风险回复（闲聊、简单事实查询、用户明确知道 Agent 可能不准确的场景），跳过验证器。对高风险回复（医疗、金融、法律、涉及人身安全建议），启用双验证器。

这是一个**风险分级验证策略**——又是一个该讲中谷歌「委员会拍板」的映射：不是每个候选人都需要委员会激烈辩论，但对边缘案例和关键岗位，委员会介入。

---

#### 原则 7：事前验尸 —— 在操作执行之前假设它已经失败

心理学家 Gary Klein 提出了一个极其廉价但有效的决策工具：**事前验尸。**

操作：在决策**执行之前**，召集相关人员，假设现在已经是未来，这个决策**已经惨败**。每人独立写下「为什么会失败」的故事。

它的机制是多重的：
- **对抗确认偏误**：决策一旦形成，人会不自觉搜集支持性证据。事前验尸强制换到对立视角。
- **安全释放异议**：在正式会议上说「我反对」有社交成本。在「假装已经失败，分析原因」的游戏框架中，表达顾虑变得安全——你不是在反对决策，你是在扮演侦探。
- **系统性扫描盲区**：多人多角度生成失败叙事，覆盖单一个体想象力的局限。

在 Agent 架构中，事前验尸可以这样实现：

对于**关键操作**（调用支付 API、修改数据库、发送对外邮件、给出医疗/法律建议），在执行之前，用一个独立 LLM 实例运行：

```
[Premortem Prompt]
假设现在已经是未来。以下操作已经执行，并且导致了负面后果。

**即将执行的操作**：
{操作内容和参数}

**当前上下文**：
{用户 query 摘要、相关历史、触发该操作的原因}

**任务**：
生成三个独立且具体（非笼统）的失败场景。每个场景需要包含：
1. 失败的直接原因（机制层面，不是「运气不好」）
2. 当时有什么信号被忽略了，事后看是明显的警告
3. 如果当时注意到了这个信号，可以采取什么不同的操作

约束：
- 不要生成笼统的「信息不准确」——要具体：哪条信息？为什么不准确？
- 不要生成「用户不满意」——要具体：用户预期是什么？操作如何违反了它？
```

然后，对这些失败场景做严重性评分。如果任一场景的**严重性 × 可预防性**超过阈值，触发：
- 人工确认（高风险操作）
- 自动降级（用更保守但更安全的替代操作）
- 附加约束后重新执行（加上事前验尸发现的检查项）

**这不是让 Agent 「更谨慎」。** 这是用结构化流程强制引入被乐观偏差压制的信号。

为什么 Agent 需要这个？因为 LLM 在生成操作计划时，本身就处于「论证这个计划是合理的」的推理模式中（它刚刚生成了计划，推理链条一路向前）。让它自己再检查一遍自己的计划，它仍然处于同一个推理模式——只是在同一个方向上多走几步。事前验尸强制它切换方向：**假设最坏情况，从那里往回推导。**

---

#### 原则 8：约束外化 —— prompt 里的规范不是规范，代码里的才是

回顾决策卫生的核心隐喻：你不在手术室门口贴「请医生记得洗手」，而是设计一个流程，让医生**不洗手就进不去**。

在 Agent 架构中：

| Prompt 中的「软约束」                                  | 架构中的「硬约束」                                           |
| ------------------------------------------------------ | ------------------------------------------------------------ |
| 「请确保回复中的每条事实都能被检索结果支持」           | Verifier 逐条交叉比对 → 不通过 → 退回重生成或降级回复        |
| 「如果用户问题涉及医疗建议，请务必谨慎并建议咨询医生」 | Router 识别医疗 query → 强制追加免责声明到回复前缀 + 限制可调用工具集 |
| 「保持回复简洁，不超过200字」                          | 后处理阶段：字数统计 → 超限触发摘要压缩 → 压缩后再校验关键信息是否保留 |
| 「不要对敏感政治话题发表立场」                         | Router 识别 → 直接返回预设的安全回复，不进入生成流程         |
| 「如果不知道答案，诚实地承认」                         | Verifier 检查回复中是否包含不确定表达 → 如果没有但事实支撑不足 → 自动追加不确定性声明 |

**为什么「写在 prompt 里」不够？**

因为 LLM 每生成一个 token，都是在进行一场注意力竞争。prompt 中有角色描述、格式要求、few-shot 示例、用户 query、对话历史、工具返回结果……「请保持回复不超过200字」只是数百条约束中的一条。在注意力竞争中，它有时候赢（Agent 回复确实简洁），有时候输（Agent 突然开始长篇大论）。

而架构约束不参与注意力竞争。它在 LLM 之外执行——在 token 生成完成之后截断、在 LLM 调用之前拦截、在独立的模块中校验。**它不需要模型「记住」任何东西。它直接物理性地阻止违反约束的行为发生。**

一个简单的测试：如果你可以把一个约束从 prompt 中删除，同时在代码中实现等价约束——删除 prompt 版本，保留代码版本。结果几乎总是更好（要么质量不变但 prompt 更短、噪声更低，要么质量提升）。

---

### 原则 9-10：训练与演进

这四条解决 Agent 上线后如何持续改进的问题——不是靠改 prompt，而是靠结构化的反馈循环。

---

#### 原则 9：影子模式 + 渐进放权 —— 避免冷启动的机械预测失效

回顾该讲的核心发现：机械预测（统计公式）需要历史数据。没有数据时，你只能靠临床判断。

新上线的 Agent 就是冷启动的机械预测系统。你的 L2 分类器还没有标注数据，L3 的阈值还没有校准，你甚至不知道哪些决策点最容易出噪声。

**影子模式**是最低风险的冷启动策略：

```
Phase 1: 影子模式（1-4周）
  - Agent 做出所有决策，但不实际执行
  - 日志记录：「如果是正式模式，Agent 会选择 X」
  - 同时：更保守的规则引擎（或人工）实际执行
  - 目的：积累决策日志，发现噪声模式

Phase 2: 低风险放权（第3-8周）
  - 对已验证的低风险决策点（如搜索结果排序），切换到 Agent 决策
  - 高风险决策点（如支付、对外沟通）维持影子模式
  - 持续监控：放权后的决策质量是否与影子模式期间一致

Phase 3: 渐进扩展（持续）
  - 每次扩展一个决策点
  - 扩展前：分析影子模式日志，确认该决策点的噪声在可接受范围内
  - 扩展后：持续监控，设置回退阈值（一旦异常率超过 X，自动回退到上一阶段）
```

影子模式的成本是延迟和资源，但它买回来的是**决策基线在可控条件下建立**。没有这个阶段，你就是在没有仪表盘的情况下盲开——你不知道 Agent 的「正常行为」是什么，所以你也无法判断什么时候「异常」发生了。

---

#### 原则 10：校准监控 —— 不看自述置信度，看行为统计

LLM 可以在被要求时输出一个置信度数值：「我 85% 确定这个答案是准确的。」这个数值本身的可靠性如何？

答案是：**不太可靠。** LLM 的置信度自述受限于它对自己知识边界的元认知能力——而这个能力本身也有噪声。

更重要的是，Agent 的大部分决策根本不会输出显式的置信度。Router 判断「需要搜索」，你不会问它「你有多确定需要搜索」——你直接让它输出 true/false。但这个 true/false 背后有一个隐式的置信度分布，而这个分布的健康程度只能通过**统计行为**来监控，不能通过**自述**来评估。

需要持续监控的指标：

| 指标         | 含义                                        | 异常信号                                                     |
| ------------ | ------------------------------------------- | ------------------------------------------------------------ |
| 路由准确率   | Router 判断是否需要搜索，与事后标注的一致性 | 准确率突然下降 → 可能模型版本更新改变了路由行为基线          |
| 工具选择分布 | 各类工具被调用的频率分布                    | 某类工具的调用率突然飙升/骤降 → prompt injection 或 query 分布漂移 |
| 追问率       | Agent 选择追问而非直接回复的比例            | 突然升高 → Agent 可能变「懒」了，该直接回答时选择追问        |
| 拒绝率       | Agent 拒绝回复的比例（安全规则触发）        | 突然升高 → 安全规则可能过于敏感，过度拒绝合法请求            |
| 评估分数分布 | Evaluator 对搜索结果打分的均值/方差         | 分数均值突然上移 → 评估标准在漂移；方差突然增大 → 评估者噪声增加 |
| 延迟分布     | 端到端响应时间、各模块耗时                  | 某个模块耗时突然增加 → 可能陷入搜索循环，或下游 API 异常     |

这些不是「用户满意度」指标。它们是**决策过程本身的健康指标**。用户满意度的变化可能来自很多因素（产品改了、竞品变了、用户群体变了）。但校准监控告诉你的是：**你的 Agent 的判断系统本身是否在正常运转。**

一个实际场景：你升级了 GPT-4 到 GPT-4.5。用户反馈没有明显变化。但校准监控显示，Router 的 `need_search=true` 比例从 35% 降到了 22%。这意味着新模型对自身知识的边界判断更自信了——或者更不准确了。你不知道答案，但你知道了问题。你可以针对性地标注一批样本来验证，而不是等用户投诉。

### 原则 11-12：元原则

最后两条不是给 Agent 的设计约束，而是给**设计 Agent 的人**的设计约束。

#### 原则 11：损失函数显式化 —— 「宁可错杀还是宁可错放」是你的选择，不是技术参数

贝叶斯决策理论中，损失函数 \(L(a, \theta)\) 定义：当真实状态为 \(\theta\) 时，采取行动 \(a\) 带来的损失。最优决策是选择使期望损失最小的行动。

框架告诉你如何最小化损失。但**损失函数本身是什么——这是规范性问题，不是技术问题。**

以 Agent 的内容安全为例：

- **方案A**（宁可错杀）：任何疑似敏感话题 → 直接拒绝回复。误拒了一些合法但边界模糊的请求，但漏放风险极低。
- **方案B**（宁可错放）：仅在明显违规时拒绝，模糊地带放行。漏放风险较高，但误拒率低。
- **方案C**（分级处理）：模糊地带先给出谨慎回复+安全提示，同时后台标记人工复查。

这三个方案背后是不同的损失函数。谁的损失？用户被误拒的损失 vs 平台被处罚的损失 vs 社会被有害信息影响的损失。这些损失的权重分配没有一个数学公式可以推导。它是**你的价值选择**。

但很多 Agent 系统把这个问题推给了 LLM：「你自行判断这个回复是否安全。」这等价于——把定义损失函数的权力交给了一个训练数据来源不明、对齐策略不透明的黑箱。

**损失函数应该被显式定义，并被编码为架构规则。** 不是「LLM 你看着办」，而是「接入一个分类器/规则引擎，当置信度处于 [x, y] 区间时执行预设的处理策略」。

另一个例子：搜索 Agent 的「边际信息价值」阈值。多搜一次的成本（延迟、token）和数据质量提升的收益之间如何权衡？如果你不显式设定这个阈值，Agent 的隐式行为就是模型训练数据中「更全面」的奖励信号在起作用——而这个信号未必匹配你的产品场景。一个面向研究的 Agent 可能应该「宁可多搜」，一个面向即时客服的 Agent 应该「宁可快答」。

**关键心态**：每当你听到自己说「让模型自己判断」，停下来问——这个判断中包含价值选择吗？如果有，那个选择应该是你做的。

---

#### 原则 12：审计独立化 —— 设计流程的人不能是评估流程的人

回到该讲中法官量刑研究的最后一层含义。研究者设计了实验，评估了法官的噪声。但研究者自己的判断有没有噪声？他们设计的实验框架——「相似案件应得相似判决」的预设——本身是不是一种价值观嵌入？

同样的递归问题存在于 Agent 设计中。你设计了评估指标、打分标准、聚合权重。你的设计反映了你对「好 Agent」的理解。但如果这个评估体系本身有偏差（比如某个维度的权重过高，导致某种类型的错误被系统性低估），谁能发现？

**答案不是「更好的评估」。答案是独立审计。**

在组织层面上，这意味着：
- 设计 Agent 决策流程的团队，和评估 Agent 决策质量的团队，不应该完全重叠。
- 审计可以很简单：定期抽取一批决策日志，交给一个不参与日常开发的同事（或外部人员），让他们基于原始信息独立判断「Agent 的决策是否合理」。
- 如果审计者与 Agent 的判断经常不一致，可能不是 Agent 错了——可能是指标的定义本身就值得商榷。

在技术层面上，这意味着：
- 评估 Agent 的离线 benchmark，不能只包含 Agent 自身的设计者定义的「正确行为」。
- 应该包含**对抗性测试用例**——专门设计来暴露评估框架自身盲区的边界案例。

这不是 paranoia。这是决策科学的递归应用：如果一个方法声称能降噪，那么设计这个方法的过程本身，也应该接受同样的降噪审视。否则你只是把噪声从决策对象转移到了决策设计者身上——这不算进步。

## 8. 这套方法不能解决什么

一个声称能解决一切问题的框架是不可信的。以下是明确的能力边界。

### 8.1 不能告诉你正确的损失函数

贝叶斯框架能帮你形式化「给定损失函数，什么是最优决策」，但损失函数本身——「漏诊 vs 误诊哪个更严重」，或者「用户被误拒 vs 平台被处罚的相对权重」——这是规范性的价值选择。决策科学让你把这个选择**显式化**，而不是隐藏在模型的隐式行为中。但它不能替你做选择。

### 8.2 最优结构化粒度取决于环境稳定性

卡尼曼的机械预测（0.32）在**稳定环境**中优于临床判断（0.15）。但在快速变化的环境中，基于历史数据训练的机械预测偏差会上升（数据过时），而临床判断（直觉）可能更适应变化。

映射到 Agent：你引入的结构化流程越多，系统对环境变化的适应速度越慢。模型版本更新了、用户行为变了、下游 API 改了——你的 L2 分类器需要重新标注和训练，而如果你直接用 LLM 做综合判断，它自动适应了新环境。

这是 Gigerenzer 反击卡尼曼的核心战场：在高度不确定、快速变化的环境中，「少即是多」的简单启发式可能比严格的结构化流程更稳健。

对于 Agent 设计者来说，这意味着：**没有统一的「最优拆解粒度」。** 需要根据你的具体场景——任务稳定性、模型更新频率、错误的成本——来做权衡。

### 8.3 快速变化环境中，过多结构化可能适得其反

接上一条。当前 LLM 的能力边界在快速变化（每季度新模型、新能力）。如果你为一个模型版本精心调优了路由分类器、评估阈值、聚合权重——下一个模型版本可能让这些全部过时。

一种策略是**结构化梯度的动态调整**：在模型稳定期，增加结构化约束（降噪）；在模型快速进化期，适当放宽结构化，利用模型自身能力的提升。这类似于投资中的再平衡策略——不是设定了就不改，而是定期重新评估。

### 8.4 自我改进循环的稳定性是开放问题

如果 Agent 收集自己的决策日志，自动调整 prompt 或路由规则——那么调整后的 Agent 还是同一个 Agent 吗？用调整后的标准去评价调整前的决策，是否合理？如果改进方向是错的（过拟合到了某个阶段的用户反馈），谁来发现？

这不是拒绝自动化改进。而是说：**改进循环本身需要有外部锚点**——一个独立于被改进系统的评估框架。这类似于原则 12 在时间维度上的展开：改进机制的审计者不应该就是改进机制本身。

这个问题目前没有确定的学术答案。任何声称「Agent 可以实现完全自动化的自我优化」的承诺，都至少早产了几年。

### 8.5 不是所有 Agent 都需要这 12 条

最后，也是最需要说清楚的一点：这 12 条原则是一个**风险-结构梯度**设计菜单，不是一份必须全部照做的清单。

一个内部使用的、低风险的、用来整理文档的 Agent，可能只需要原则 1（分解）的轻量版本 + 原则 4（停止规则）。一个面向公众的、提供医疗信息的 Agent，应该启用原则 1-8 的完整版本。

决定启用多少结构化的，是你对「这个错误如果发生了，代价有多大」的评估。这个评估本身，也是一种损失函数的显式化——回到原则 11。

## 9. 结语：一种思维习惯

最后，这篇文章最重要的输出也许不是 12 条原则本身——而是原则背后的思维习惯。

当你看到 Agent 犯了一个错误，你的第一反应是什么？

大多数开发者的第一反应是「改 prompt」——加约束、加 few-shot、加更详细的指令。

但如果你读完了这篇文章，也许你的第一反应会变成一个诊断问题：

- 这是**偏差**（系统性地偏向某一类错误），还是**噪声**（不可复现的随机错误）？
- 如果是偏差，是哪一个评估维度的系统偏差？Router 对某类 query 系统性地高估了搜索需求？还是 Evaluator 对格式漂亮的文本系统性地放松了事实核查？
- 如果是噪声，是水平噪声（某个模块的整体倾向不对）、稳定模式噪声（某类特定上下文系统性地触发误判），还是情境噪声（同一输入偶尔出错）？
- 这个错误能不能通过**架构约束**来拦截，而不是通过**更好的指令**来规避？

这种思维习惯，才是决策科学留给 Agent 设计者最持久的遗产。

卡尼曼在《噪声》中写道：「哪里有判断，哪里就有噪声——而且比你想象的更多。」这句话对 LLM Agent 同样适用，甚至更加适用。因为 LLM 的噪声来源比人类更丰富、更难直接观察、更隐蔽地嵌入在注意力机制和训练分布中。

但也正因为如此，**架构设计比以往任何时候都更重要。** 当模型能力还在快速进化时，架构是你的稳定锚点。你不能控制模型提供方什么时候更新版本、训练数据中有什么新的偏误、RLHF 对齐策略引入了什么新的行为模式。但你可以控制——判断是如何被拆分的、信息是如何在被模块之间传递的、约束是如何被强制执行的、决策是如何被记录和监控的。

好架构不追求消除所有噪声。它追求让噪声**可控、可观测、可追溯**。它承认 LLM 会犯错，然后把设计重心从「阻止错误发生」转移到「当错误发生时，它被限制在哪个范围内，能被多快发现，以及修复的代价有多小」。

这就是决策卫生在 AI 时代的翻译：**不是造一个不犯错的 Agent，而是设计一个让错误难以蔓延的系统。**
