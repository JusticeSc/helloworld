# RAG 系统评估

---

## 引言

检索增强生成（Retrieval-Augmented Generation, RAG）已经成为大模型落地的主流范式。但一个尴尬的事实是：大多数组件都有成熟的评估方案——搜索有 NDCG，翻译有 BLEU，摘要有 ROUGE——唯独 RAG 作为一个检索+生成的混合系统，长期缺乏专属的评估语言。

RAGAS、ARES、TruLens 等框架的出现填补了这个空白。但工具只是表象，真正值得追问的是：**这些指标是怎么建构出来的？它们的工作原理是什么？它们如何指导工程决策？**

本文试图从**观念层→逻辑层→操作层→诊断层**四个递进的切面，把 RAG 评估这件事讲透。

---

## 一、观念层：指标的建构方法论

### 1.1 从失败模式反推

RAG 评估指标的建构起点不是"什么是好"，而是**"什么是坏"**。这是建构论而非本质论的方法：

```
观察到的失败模式               →  评估构念
─────────────────────────────────────────
检索返回无关文档                  Context Relevance
检索遗漏关键文档                  Context Recall
答案编造不存在的事实              Faithfulness
答案偏离用户问题                  Answer Relevance
答案正确但无法溯源                Attribution
```

**核心原则**：每个指标必须对应一个明确的失败模式。如果一个指标不能映射到任何已知的失败模式，它很可能是在测量噪声。

### 1.2 原子化：从模糊直觉到可验证单元

Faithfulness 是理解这一原则的最佳案例。最朴素的直觉是"答案不能瞎编"，但如何让机器判断"瞎编"？

RAGAS 的选择是**原子化拆解**：

```
整体打分路径（不可解释）:
  答案 → LLM → "忠实度: 0.7"  → 为什么是 0.7？哪部分有问题？不可知

原子化路径（可解释）:
  答案 → 拆成 5 条原子陈述 → 逐条在上下文中验证
  → 3 条有依据，1 条矛盾，1 条无信息
  → Faithfulness = 3/5 = 0.6
  → 具体哪条出了问题，一清二楚
```

这条原则贯穿 RAG 评估的全部设计：**复合构念必须分解到不可再分的测量单元，否则分数缺乏解释力**。

### 1.3 评估三角：向量而非标量

RAG 质量不是单一数值，而是由三元关系构成的向量：

```
              Question
                 ▲
                / \
               /   \
              /     \
   Context ◄─────────► Answer
   Relevance          Faithfulness

  + Answer ↔ Question = Answer Relevance
```

三条边缺一不可。即使上下文相关、答案忠实，如果答案完全偏离问题，用户仍然不满意。Faithfulness 高不能弥补 Answer Relevance 低——**不同维度的质量不可相互补偿**。

### 1.4 评估系统自身的效度问题

指标建构面临一个根本性挑战：**用 LLM 评估 LLM 的输出，是否构成循环？**

RAGAS 的应对策略：
- 推荐使用比被评估模型更强的评估模型（如 GPT-4 评估 GPT-3.5）
- 定期用人工标注做 Cohen's κ 校准（κ ≥ 0.6 视为可用，κ ≥ 0.8 视为可信）
- 明确声明评估的置信边界，不假装"精准"

**指标建构者必须诚实地报告盲区。不存在完美的指标，只存在已知局限的指标。**

## 二、逻辑层：核心指标的工作原理

### 2.1 Faithfulness：最关键的 RAG 指标

Faithfulness 回答一个问题：**答案中的每一句话，能不能在检索到的上下文中找到出处？**

执行流程分三阶段：

**阶段 1：声明提取**

LLM 将答案拆解为最小事实单元：

```
答案: "Python 由 Guido 于 1991 年创建，是解释型语言。"
  ↓ LLM 拆解
[
  "Python 由 Guido van Rossum 创建",
  "Python 于 1991 年创建", 
  "Python 是解释型语言"
]
```

**阶段 2：逐条验证**

每条声明独立与上下文比对。关键设计：**验证时不提供原始问题**，防止 LLM 用自己的知识"脑补"。

```
上下文:
  [1] "Python 由 Guido van Rossum 创建"
  [2] "Python 支持多种编程范式"

验证语句 1: "Python 由 Guido van Rossum 创建"
  判定: YES（chunk1 明确支撑）

验证语句 2: "Python 于 1991 年创建"
  判定: IDK（上下文未提及年份）

验证语句 3: "Python 是解释型语言"
  判定: IDK（上下文未提及语言类型）
```

**阶段 3：计算**

```
Faithfulness = |YES| / |Total| = 1/3 = 0.333
```

IDK 为什么算作不忠实？这源于 RAGAS 的哲学立场：**RAG 是检索绑定的——模型应基于给定上下文回答，而非依赖内部知识**。即使"1991 年"是事实，只要上下文没说，输出它就是"不忠实于检索结果"。

### 2.2 Answer Relevance：反向推理的巧妙设计

直接测量"答案切题吗"很难，但可以做逻辑等价替换：**如果答案切题，那么从答案出发应该能反向推出原始问题**。

```
答案 → LLM → 生成该答案可能回答的 3 个问题
原始问题: "Python 的并发模型是什么？"
生成问题: "Python 的 GIL 对多线程有什么影响？"
          "Python 中 CPU 密集任务应该用什么方案？"
          "Python 的 asyncio 适用于什么场景？"

Answer Relevance = mean(cos_sim(Embed(原始问题), Embed(生成问题)))
```

这是一个反绎推理（Abduction）的经典应用：当构念无法直接测量时，通过"如果 A 成立则 B 必然成立"的逻辑链，转而测量 B。

### 2.3 Context Relevance 与检索指标群

Context Relevance 在句子级粒度评估检索质量：

```
上下文中的总句子: 5 句
与问题相关的句子: 2 句
Context Relevance = 2/5 = 0.4
```

它与传统检索指标形成互补：

| 指标              | 测量内容                   | 需要 GT？ |
| ----------------- | -------------------------- | :-------: |
| Context Relevance | 检索噪声比例               |    否     |
| Context Precision | 相关文档的排序质量         |    是     |
| Context Recall    | GT 信息的覆盖完整度        |    是     |
| Recall@K          | Top-K 中命中相关文档的比例 |    是     |
| MRR               | 第一个相关文档的排名       |    是     |
| NDCG              | 排序质量的加权评分         |    是     |

**关键设计**：RAGAS 核心三指标（Faithfulness、Answer Relevance、Context Relevance）不需要 Ground Truth。这并非巧合——写 GT 是评估中最昂贵、最不可扩展的环节。检索上下文被设计为"代理 GT"。

## 三、操作层：工程实现与成本

### 3.1 RAGAS 的并行调度

```
Level 0: 无依赖，并行执行
  ├─ faithfulness（需要 answer + contexts）
  ├─ answer_relevancy（需要 question + answer）
  └─ context_relevancy（需要 question + contexts）

Level 1: 需要 GT，在 Level 0 之后执行
  ├─ context_precision
  └─ context_recall
```

### 3.2 单样本评估成本

以 GPT-4o 为例，单样本三指标全跑的 LLM 调用统计：

```
Faithfulness:         1 次分解 + 3~8 次验证 = 4~9 次调用
Answer Relevance:     1 次反向生成 = 1 次调用
Context Relevance:    1 次句子提取 = 1 次调用
─────────────────────────────────────────
合计: ~6~11 次 LLM 调用/样本
成本: ~$0.02/样本（三指标全跑）
```

1000 条评测集约 $20。这就是为什么 RAGAS 被描述为"可规模化"——它用 LLM 替代了人工标注，成本降低了几个数量级。

### 3.3 评估流水线的工程化

```
评测集构建要求:
  □ 覆盖 6+ query 类型（单跳/多跳/对比/时间/过程/歧义）
  □ 每条含 question、answer、contexts、ground_truth
  □ 样本量 ≥ 50（小版本）/ ≥ 200（正式版本）
  □ 至少 20% 样本经过双人交叉验证

CI/CD 集成:
  □ 每次 PR 自动触发评估
  □ 未达阈值时阻止合并
  □ 评估报告持久化，可追溯历史趋势

在线监控:
  □ P50/P95/P99 延迟
  □ TTFT（首 token 延迟）
  □ 用户点赞率 / 重复提问率 / 复制率
  □ 每周抽样人工评估，校准离线指标
```

## 四、诊断层：指标如何指导实践

指标的价值最终体现为**诊断能力**。这需要理解指标之间的关联模式。

### 4.1 四条诊断象限

```
                    Context Relevance
                    低 ←─────────→ 高
            ┌─────────────────────────────┐
      高    │  象限 II: 罕见               │  象限 I: 健康型
            │  上下文差但答案忠实           │  上下文相关 + 答案忠实
  Faithful  │  → 检查评估是否准确           │  → 继续监控
            │                              │
      低    │  象限 III: 双败型            │  象限 IV: 生成故障型
            │  检索+生成都坏了             │  上下文相关但答案不忠实
            │  → 先修检索，再修生成         │  → 修改 Prompt，降低幻觉
            └─────────────────────────────┘
```

### 4.2 诊断决策树

```
Faithfulness 低
  │
  ├─ Context Relevance 也低？
  │   └─ YES → 先修检索
  │       修好后重新测
  │
  └─ Context Relevance 高？
      └─ 问题在生成端，分析幻觉类型:
    
          A. 知识嫁接型 → 强化"逐句标注来源"
          B. 过度推断型 → 约束"只陈述上下文明确提到的"
          C. 格式冲突型 → 对数字/实体加特殊约束
          D. 自带知识型 → 降低 temperature，强调上下文优先
```

### 4.3 检索修复指南

```
Context Relevance < 0.5（噪声大）:
  1. 换嵌入模型（通用 → 领域微调）
  2. 加 Cross-Encoder Reranker
  3. 优化 Chunk 策略（大小、重叠、语义分块）
  4. Query 改写（LLM 预处理用户问题）

Context Recall < 0.6（遗漏多）:
  1. 增加 K 值（检查 Recall@K 曲线是否饱和）
  2. 混合检索（BM25 + 向量，RRF 融合）
  3. 多路召回（异构数据源的并行检索）
  4. 父文档检索（Chunk → 完整文档）

Context Precision 远低于 Precision@K（排序差）:
  1. 加 Reranker（最有效）
  2. 调整融合权重
```

### 4.4 生成修复优先级

```
Level 1: Prompt 工程（零延迟、零费用）
  → 加入引用约束、temperature=0
  → 通常提升 Faithfulness 0.10-0.20

Level 2: 约束生成参数
  → temperature=0, top_p=0.5
  → 减少随机性驱动的编造

Level 3: 后验证 + 重生成
  → 检测到幻觉 → 自动重新生成
  → 延迟翻倍，但 Faithfulness 可提升 0.15-0.30

Level 4: 换模型
  → GPT-3.5 → GPT-4: 提升 0.10-0.15
  → 但成本增加 10-20 倍

Level 5: 领域微调
  → 高投入高回报，需持续维护
```

## 五、扩展维度：被忽视的评估方向

### 5.1 幻觉检测的分层

Faithfulness 测量的是"检索绑定度"，但幻觉还可以进一步分解：

```
忠实度幻觉（与上下文矛盾）:
  A. 事实矛盾: Context 说 1991，Answer 说 1995
  B. 无中生有: Context 没有相关信息，Answer 凭空编造

事实性幻觉（与世界知识矛盾）:
  答案可能忠实于上下文，但上下文本身就是错的
  → 需要外部知识库验证（FactScore 等方法）

检测方法:
  NLI 模型: 前提(上下文) → 假设(声明)，判蕴含/矛盾/中性
  SelfCheckGPT: 同一 prompt 多次采样，交叉比对一致性
  Token 概率: 低概率 token 可能是幻觉信号
```

### 5.2 归因精度

比 Faithfulness 更严格——不仅要求每句话有出处，还要求**出处引用正确**：

```
声明: "Python 2 于 2020 年停止维护"
系统引用: Chunk #5

验证: Chunk #5 是否确实包含这一信息？
  是 → 正确归因
  否 → 归因错误（即使声明本身正确）
```

归因错误暴露了检索片段相关信息但不完整，或模型"嫁接"了不同来源的信息。

### 5.3 多轮对话评估

单轮 RAG 指标只能测量孤立的 (Q,A) 对。多轮对话需要额外维度：

| 指标                      | 含义                             |
| ------------------------- | -------------------------------- |
| Context Reuse Rate        | 后续轮次是否有效利用前序上下文   |
| Topic Drift Score         | 对话偏离原始话题的程度           |
| Contradiction Rate        | 跨轮次自相矛盾的频率             |
| Resolution Rate           | 用户问题是否在多轮内解决         |
| Hallucination Propagation | 早期轮次的幻觉被后续轮次引用放大 |

### 5.4 效率与业务指标

离线质量好≠用户体验好。最终需要在线指标做终审：

```
效率维度:
  - P50/P95/P99 端到端延迟
  - TTFT（Time To First Token）
  - Cost per Query
  - Context Efficiency（单位相关性的 token 成本）

业务维度:
  - 点赞率 / 复制率 / 分享率
  - 重复提问率（答案是否一次解决）
  - 人工转接率（客服场景）
  - Task Completion Rate（任务型场景）
```

### 5.5 噪声鲁棒性

```
Noise Robustness = 含噪上下文下的性能 / 干净上下文下的性能

测试方法:
  在检索结果中注入 30% 不相关文档
  → 测量 Faithfulness 和 Answer Relevance 的衰减
  → 好的 RAG 系统即使在噪声中也能保持稳定

反事实鲁棒性:
  在正确上下文中插入虚假信息
  → 理想表现: 模型能够识别并忽略
```

## 六、建构原则总结

贯穿 RAG 评估指标设计的是八条核心原则：

```
原则 1: 失败驱动
  指标从已知失败模式反推，而非从抽象"好"的定义正推

原则 2: 一一映射
  每个指标对应一个明确的失败模式，不可互相替代

原则 3: 原子化
  复合构念必须分解到最小可验证单元

原则 4: 反绎推理
  当直接测量不可行时，通过逻辑等价链间接测量

原则 5: 工具透明
  明确声明测量工具的偏差，量化并纳入解读

原则 6: 测量域约束
  测量时限定在给定信息域，防止域外信息污染

原则 7: 不可约性
  不同维度的质量不可相互补偿，保留向量而非压缩为标量

原则 8: 效度自反
  指标自身需要持续验证，诚实地报告盲区
```

## 七、最小可行评估方案

对于即将上线或刚上线的 RAG 系统，不需要所有指标一步到位。推荐分阶段递进：

```
第一阶段（上线前）:
  RAGAS 三指标: Faithfulness + Answer Relevance + Context Relevance
  + Recall@5 + MRR
  目标: 确保检索和生成的基本质量

第二阶段（灰度期间）:
  + Context Recall（如果标注了 GT）
  + 幻觉分类检测（知识嫁接 / 过度推断 / 自带知识）
  + P50/P95 延迟监控
  目标: 定位具体故障模式

第三阶段（全量后）:
  + 在线信号: 点赞率、重复提问率、人工转接率
  + 多轮评估（如果是对话系统）
  + 噪声鲁棒性测试
  + 月度人工抽检校准
  目标: 建立离线-在线闭环
```

## 结语

RAG 评估的本质不是一套打分公式，而是一套**诊断语言**。它的目标是让"感觉系统有问题"这种模糊感知，转化为"Faithfulness 从 0.64 提升到 0.87，因为我们在 Prompt 中加入了逐句标注来源的约束"这种可验证的工程行为。

没有指标的优化是赌博，有指标的优化是工程。而好的工程，始于知道自己在测量什么、怎么测量、以及测量结果的边界在哪里。
